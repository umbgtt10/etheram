# EtheRAM — Implemented Capabilities

This document is the canonical inventory of capabilities already implemented in the workspace. It complements [ARCHITECTURE.md](ARCHITECTURE.md), which explains the stable 3-6 model, and [ROADMAP.md](ROADMAP.md), which lists only the remaining work.

---

## Capability Matrix

| Feature Family | Implemented Capabilities | Relevant READMEs | Project-Level Value |
|---|---|---|---|
| Core 3-6 architecture | Six dimensions, three spaces, single `step()` primitive, pure protocol handling, action partitioning, observer-based visibility, swappable components through trait-driven construction | [core/README.md](../core/README.md), [etheram-node/README.md](../etheram-node/README.md), [raft-node/README.md](../raft-node/README.md), [ARCHITECTURE.md](ARCHITECTURE.md) | Establishes the main research claim: one node decomposition can remain stable while implementations, runtimes, and protocols vary |
| Etheram single-node core | `EtheramNode`, builders, context builder, partitioner, storage/cache adapters, executor, observer, execution-engine integration | [etheram-node/README.md](../etheram-node/README.md) | Provides a real reusable node substrate rather than a paper architecture or a test-only prototype |
| IBFT consensus | Static validator set, quorum `⌊2n/3⌋ + 1`, round-robin proposer, PrePrepare/Prepare/Commit flow, view change, locked-block handling, highest-round certificate precedence, future-round buffering, deduplication, replay rejection, malicious-block quarantine, validator-set updates | [etheram-node/README.md](../etheram-node/README.md), [etheram-validation/README.md](../etheram-validation/README.md), [etheram-embassy/README.md](../etheram-embassy/README.md) | Proves the architecture can host a Byzantine protocol with real safety machinery instead of only simple happy-path consensus |
| Signature abstraction and real crypto | `SignatureScheme` trait, `MockSignatureScheme`, `Ed25519SignatureScheme`, commit signatures, prepared-certificate verification, real Ed25519 in embedded and desktop paths | [etheram-node/README.md](../etheram-node/README.md), [etheram-embassy/README.md](../etheram-embassy/README.md), [etheram-node-process/README.md](../etheram-node-process/README.md) | Shows that cryptography is swappable infrastructure rather than baked into protocol code, while still supporting real signatures in deployed paths |
| Ethereum-like chain semantics | Accounts, balances, nonces, blocks, deterministic hashes, receipts, receipts root, state root placeholder hashing, genesis accounts, pending transaction handling, client queries, contract storage, block re-execution checks | [etheram-node/README.md](../etheram-node/README.md), [etheram-validation/README.md](../etheram-validation/README.md), [etheram-embassy/README.md](../etheram-embassy/README.md) | Makes EtheRAM a meaningful Ethereum-like system rather than a consensus-only skeleton |
| Execution and gas model | `TinyEvmEngine`, `ValueTransferEngine`, `NoOpExecutionEngine`, opcode gas metering, intrinsic gas, out-of-gas semantics, block gas limit enforcement, deterministic transaction ordering, transaction pool prioritization | [etheram-node/README.md](../etheram-node/README.md), [etheram-validation/README.md](../etheram-validation/README.md) | Demonstrates that the architecture can carry real execution semantics and not just message-passing consensus |
| Persistence and recovery | `ConsensusWal`, `WalWriter`, protocol restoration from WAL, persisted replay state, locked-block recovery, storage-backed restart continuity | [etheram-node/README.md](../etheram-node/README.md), [etheram-validation/README.md](../etheram-validation/README.md), [etheram-embassy/README.md](../etheram-embassy/README.md), [etheram-node-process/README.md](../etheram-node-process/README.md) | Strengthens the claim that the decomposition survives failures and restarts, not just clean in-memory runs |
| Stage 2 cluster validation | Shared-memory multi-node cluster harness, Byzantine injection, view-change scenarios, persistence/restart scenarios, validator-update scenarios, execution-engine comparisons | [etheram-validation/README.md](../etheram-validation/README.md), [raft-validation/README.md](../raft-validation/README.md) | Turns the architecture into a testable distributed system framework instead of a single-node library |
| Embedded `no_std` deployment | Embassy-based ARM Cortex-M4 runtime, channel and UDP transports, in-memory and semihosting storage, two end-to-end configurations, multi-act QEMU scenario, graceful shutdown | [embassy-core/README.md](../embassy-core/README.md), [etheram-embassy/README.md](../etheram-embassy/README.md), [raft-embassy/README.md](../raft-embassy/README.md) | Proves the same architecture survives constrained `no_std` deployment rather than only desktop/server environments |
| Desktop multi-process deployment | `etheram-node-process`, gRPC peer transport, gRPC external interface, sled persistence, file-backed WAL, fleet TOML configuration, native launcher, dashboard, partition/heal controls | [etheram-node-process/README.md](../etheram-node-process/README.md), [etheram-desktop/README.md](../etheram-desktop/README.md) | Proves the architecture also survives real OS-process boundaries, real networking, and operator-facing workflows |
| Raft protocol family | Independent `raft-*` crate family, pre-vote, election, log replication, snapshots, client commands and queries, state machine integration, cluster tests, embedded scenario | [raft-node/README.md](../raft-node/README.md), [raft-validation/README.md](../raft-validation/README.md), [raft-embassy/README.md](../raft-embassy/README.md) | Validates generality across a second consensus family with different safety assumptions and control flow |
| Testing and quality gates | 1,029 automated tests, protocol-level integration tests, cluster-level validation, desktop/process tests, QEMU scenarios, docs/test/app gates, mandatory `no_std` checks | [README.md](../README.md) | Makes the project credible as an executable research artifact rather than an aspirational design |
| Observability and operator surface | `Observer` / `RaftObserver`, semihosting observer, dashboard-fed launcher state, client-facing RPCs, partition controls, shutdown controls | [etheram-node/README.md](../etheram-node/README.md), [raft-node/README.md](../raft-node/README.md), [etheram-embassy/README.md](../etheram-embassy/README.md), [etheram-desktop/README.md](../etheram-desktop/README.md), [etheram-node-process/README.md](../etheram-node-process/README.md) | Shows the decomposition supports debugging, inspection, and operations rather than only internal protocol correctness |
